# Newsletter editorial outcome eval

`bash skills/01-newsletter-editorial/eval/run-eval.sh` is the deterministic CI layer. It uses synthetic, held-out fixtures and the Python standard library only. Static contract mutation checks and outcome validation are deliberately separate: the outcome validator compares candidate decisions, reason codes, and next actions to frozen expected results rather than checking whether the skill text was loaded. It also validates the versioned local runner-output interface; this is an interface test, not evidence of model or skill performance.

The manually dispatched harness runs `run_harness.py` with a repository-controlled runner, declared runner version, declared versioned model, and digest-pinned approved image. It creates a new temporary workspace for every condition and trial, exposing only opaque case IDs, prompts, the runner, and (when enabled) `SKILL.md`; expected outcomes, semantic case names, and candidate answers remain evaluator-only. The runner receives no condition or trial signal: the host adds those fields after parsing its output, so the only ablation difference is `SKILL.md` presence. Each Docker execution uses a read-only root filesystem and `--network none`, without mounting the checkout or providing credentials. The runner must emit JSON records with `case_id`, `outcome`, `runner_version`, and `model`; values must match the declared run and are retained with the result.

Run three to six trials per case. The gate requires at least an 80% enabled pass rate and a positive enabled-versus-disabled delta. The manual GitHub Actions workflow retains the JSON comparison report; do not run this harness in pull-request CI. No target-model result is checked in: until an operator supplies a versioned runner, model, and digest-pinned image, the repository has no empirical enabled-versus-disabled outcome claim.
